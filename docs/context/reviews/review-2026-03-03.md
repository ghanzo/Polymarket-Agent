## CRITICAL Issues (1)

### C1. Force-cycle permanently disables simulation loop
**File**: `dashboard.py:441` | **Category**: Infrastructure

The `/api/cycle` endpoint creates a fire-and-forget `asyncio.create_task` that never resets `sim_state["status"]` back to `"waiting"` on completion. After one forced cycle, `simulation_loop` sees status `"running"` forever and skips all future cycles, logging "Previous cycle still running, skipping" indefinitely.

If the forced cycle also throws an exception, the task exception is silently swallowed by asyncio, compounding the problem.

**Fix**: Wrap the task so `sim_state["status"]` resets to `"waiting"` in a `finally` block.

## HIGH Issues (10)

### H1. Longshot bias correction is backwards for NO bets on low-midpoint markets
**File**: `simulator.py:156-162` | **Category**: Financial

When `midpoint < 0.15` (YES is a longshot), the code reduces `est_prob` (YES probability). But for a NO bet, this *increases* the implicit NO probability, making the NO bet MORE attractive — the opposite of what longshot bias correction should do. At the default 10% adjustment, this can meaningfully affect bet sizing.

**No test covers the interaction between longshot correction and NO-side betting.**

### H2. EXITED bets excluded from performance reviews, calibration, and Platt scaling
**File**: `db.py:464` | **Category**: Financial

`get_resolved_bets()` only returns bets with status `WON` or `LOST`, excluding `EXITED` (stop-loss, take-profit, stale exits). This creates survivorship bias in the learning pipeline — the calibration data, Platt scaling model, and model weights are all trained on a non-representative sample that excludes early exits.

### H3. Ensemble averages probability from only majority-side voters
**File**: `analyzer.py:689-694` | **Category**: Analysis

The ensemble excludes dissenting models from the probability average. If Model A says BUY_YES at 0.90 and Model B says BUY_NO at 0.20, the final probability is 0.90 (only Model A), not a weighted blend. This systematically inflates confidence and oversizes bets via Kelly criterion.

### H4. Thread-unsafe global caches in learning.py
**File**: `learning.py:17-19, 265-267` | **Category**: Analysis

`_cached_model_weights`, `_cached_category_weights`, and `_cached_platt_params` are module-level dicts read/written from concurrent threads (debate mode uses ThreadPoolExecutor). No locks protect these. Could produce `RuntimeError: dictionary changed size during iteration` or stale reads.

### H5. Rebuttal JSON parse failure crashes silently
**File**: `analyzer.py:457-468` | **Category**: Analysis

If a rebuttal response contains `{ invalid json }`, the guard for missing delimiters passes but `json.loads()` throws uncaught `JSONDecodeError`. The model's entire Round 2 contribution is silently dropped in debate mode.

### H6. Silent `except Exception: pass` pattern throughout pipeline
**File**: Multiple (analyzer.py:277, cycle_runner.py:66, learning.py:47, etc.) | **Category**: Analysis

Persistent failures (DB down, expired API key, broken import) are retried every cycle but never escalate. The system can run for hours in a degraded state (no calibration, no weights, no error patterns) with no clear indication.

### H7. No API retry logic on transient failures
**File**: `api.py:38-49` | **Category**: Infrastructure

`_gamma()` and `_clob()` make a single HTTP request with no retry on 429, 502, 503, or timeouts. With 10 concurrent enrichment threads hitting 200-300 API calls per cycle, transient failures silently degrade data quality.

### H8. httpx client never closed — connection leak
**File**: `cycle_runner.py:73`, `backtester.py:296,600` | **Category**: Infrastructure

`PolymarketAPI()` creates an `httpx.Client` with connection pooling (max 20) every cycle. The client is never `close()`-d. Over 24 hours of dashboard operation (288 cycles), this leaks up to 5,760 TCP connections, eventually crashing with `OSError: Too many open files`.

### H9. Unauthenticated state-modifying dashboard endpoints
**File**: `dashboard.py:419-468` | **Category**: Infrastructure

POST endpoints `/api/settings`, `/api/cycle`, `/api/close-position/{bet_id}`, `/api/close-all` have no authentication. Anyone with network access can change all trading parameters, force cycles, or close all positions.

### H10. Force-cycle exception handling — task exception silently swallowed
**File**: `dashboard.py:441` | **Category**: Infrastructure

Related to C1. If `_run_cycle` throws during a forced cycle, asyncio swallows the exception ("Task exception was never retrieved"). Status stays stuck at "running" permanently.

## Test Suite Assessment

**Grade: B-**

### Systemic Weaknesses

**1. Reimplemented Logic Tests (~20% of suite)**
Tests duplicate production logic inline and assert against the copy, never calling actual production functions. Found in: `test_profitability.py`, `test_data_integrity.py`, `test_phase3.py`, `test_cost_ops.py`, `test_integration.py`.

Example: `test_bet_won_pnl` reimplements `pnl -= pnl * config.SIM_FEE_RATE` locally instead of calling `db.resolve_bet()`. The actual fee bug goes undetected because the test tests its own copy.

**2. Vacuous Assertions**
`test_integration.py:258` — `test_disagreement_leads_to_skip` asserts `recommendation in (SKIP, BUY_YES, BUY_NO)` — accepts ALL possible values. Cannot fail.

**3. Source Code Inspection Tests**
Multiple tests use `inspect.getsource()` to check for string presence (e.g., `"SIM_FEE_RATE" in source`). These break on refactoring and give false confidence — the string being present doesn't mean it's used correctly.

**4. Over-Mocking of Database Layer**
All 17 tests in `test_data_integrity.py` mock the DB connection. They verify SQL string patterns (`assert "FOR UPDATE" in sql`) but never execute actual SQL. Every DB-related bug (fee accounting, make_interval, constraint violations) escaped because of this.

**5. Missing Edge Cases**
No tests for: NaN/negative/Infinity inputs, partial market resolution prices, entry prices near 1.0, concurrent bet placement against balance limits.

### Strengths
- Pure function tests (kelly_size, slippage, strategies) are thorough
- Proper freezegun usage in scanner and analyzer time tests
- Good `_parse_response` edge case coverage (bad JSON, missing fields, clamping)
- autouse fixture pattern in test_learning.py for state cleanup
